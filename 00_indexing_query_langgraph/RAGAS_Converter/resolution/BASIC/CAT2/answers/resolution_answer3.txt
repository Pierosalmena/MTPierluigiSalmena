What the records document you should check first (diagnostic steps drawn from prior tickets)
1) Confirm SharePoint folder permission model and which group the ACL uses
   - Check whether the folder has Unique Permissions and which group is listed in the SharePoint permissions UI (example: “SP_Compliance_Main”) [Data: Sources (30)].

2) Check Sensitivity Label / AIP encryption settings for that folder
   - Verify whether a Sensitivity Label with encryption is applied, and which Azure AD group is granted the AIP encryption key (example problem: label granted only to “Compliance_Core” while SharePoint ACL used “COMPLIANCE_TEAM”) [Data: Sources (30, 31)].
   - Verify whether the label was applied after the folder received Unique Permissions (records show labels applied after renaming/permission change caused the issue) [Data: Sources (30)].

3) Inspect sign-in tokens / claims for affected users
   - Review Azure AD Sign‑In Logs for affected users to see which group claims are present in the token (records showed tokens containing COMPLIANCE_TEAM but not Compliance_Core for affected users) [Data: Sources (30)].

4) Reproduce access behavior with minimal caching
   - Test access in Incognito/private browser and by signing out of all Office apps (records show Incognito/signed‑out sessions sometimes allowed access temporarily) [Data: Sources (30, 31)].

5) Check for group origin / Azure AD Connect effects (if externals are affected)
   - In Entra/Azure AD, check the “source” attribute for users and groups (records found some groups/users “source = Windows Server AD” vs “Azure AD”) and whether two groups with identical names but different object IDs exist (Cloud‑native vs synchronized On‑Prem) [Data: Sources (13)].

6) Check Conditional Access and device compliance policies
   - Verify CA policies that apply to SharePoint or the project resources and device compliance/Hybrid AAD join status (one case showed CA requiring compliant devices caused delays; device status “unknown” and sync delays of 6–8 hours) [Data: Sources (38)].

What the records show worked as remediation (step‑by‑step actions actually performed in prior incidents)
A. If the root cause is Sensitivity Label encryption conflicting with SharePoint ACLs
   1. Remove the Sensitivity Label temporarily from the folder to stop encryption gating access [Data: Sources (31)].
   2. Reassign the folder permissions so the SharePoint ACL points to the correct group object (align group references used by SharePoint ACL with the group granted the AIP encryption key) [Data: Sources (31)].
   3. Reapply the Sensitivity Label with the correct group integration (make sure the encryption grant contains the same group/object id used by SharePoint) [Data: Sources (31)].
   4. Invalidate existing authentication tokens for affected users (have users sign out of Office apps/browsers or intentionally expire tokens) and retest — this resolved the token-caching inconsistency in the documented case [Data: Sources (31)].
   5. Monitor and confirm consistent access for all user types (internal, external, guests) [Data: Sources (31)].

B. If the root cause is duplicate groups from Azure AD Connect (external guests losing access intermittently)
   1. Identify that two groups exist with the same display name but different origins/Object IDs (Cloud‑native group + On‑Prem synchronized group) [Data: Sources (13)].
   2. Consolidate group structure: remove the redundant Cloud group (or the redundant On‑Prem group) so SharePoint references the correct object id; clean the On‑Prem group as needed [Data: Sources (13)].
   3. Manage external guests as Cloud‑only members (so they are present in the final effective group) [Data: Sources (13)].
   4. Perform an Azure AD Connect full synchronization to ensure cloud and on‑prem state are aligned [Data: Sources (13)].
   5. Validate and monitor that external guest access is stable [Data: Sources (13)].

C. If the root cause is Conditional Access / device compliance token delays causing flickering access
   1. Identify CA policy applied to the resource (e.g., “Require compliant device for Finance Resources”) and note any recent adjustments [Data: Sources (38)].
   2. For immediate relief, adjust the CA scope (in the documented case they excluded the specific SharePoint folder from the Finance CA rule) or ensure device compliance sync completes quickly [Data: Sources (38)].
   3. Have affected users fully sign out and sign back in (clear token cache); trigger device sync for Hybrid AAD joined devices if synchronization is delayed [Data: Sources (38)].
   4. Monitor sign‑in logs to confirm Conditional Access evaluation succeeds and tokens contain updated claims [Data: Sources (38)].

D. If the root cause is incorrect group nesting
   1. Review nesting in Entra ID and correct it so the high‑level group is nested into the permission group SharePoint evaluates (example: “IT‑Projekte” was still nested into old “SP_Project_IT_v1” instead of the new “SP_Project_IT”) [Data: Sources (20)].
   2. Remove old/deactivated group, correct nesting, wait for synchronization and re-test access [Data: Sources (20)].
