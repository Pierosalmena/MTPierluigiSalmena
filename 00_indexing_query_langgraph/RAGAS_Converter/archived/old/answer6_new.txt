
# Urgent summary (TL;DR)                                                        
Most evidence points to two concurrent causes acting together: (A) a perimeter/WAF/template or TLS‑inspection change after a weekend patch that removed required header rewrites (notably X‑Forwarded‑Proto) and broke authentication/session semantics for externally routed traffic, and (B) an emergency country/geofencing control (requested after the phishing) that either was applied or attempted and is blocking non‑MFA/geo flows. Those two together explain: external login redirects and 502s, travelling legitimate users blocked, non‑browser automated accounts failing, and one integration that still works in a browser but fails in background. Treat this as a security incident and escalate to 2nd‑level (Identity + Network/WAF + Security Incident Response). [Data: Sources (15); Sources (17); Sources (4); Sources (11)].

## What likely happened (chain of events)
- Phishing delivery: two PDF files contained links to a convincing fake Microsoft login that posted credentials to algina[.]co — credential harvesting is documented and should be treated as suspected compromise for any accounts that entered credentials [Data: Sources (4)].
- Perimeter/WAF change: a security patch/firewall update over the weekend reloaded a default profile and removed the Portal‑App‑Template / custom header‑rewrite (X‑Forwarded‑Proto), causing external FQDN requests to arrive to the backend without expected headers and produce 502 Bad Gateway; disabling TLS inspection temporarily restored logins, and restoring the Portal‑App‑Template/header rewrites fixed the outage in the known incident [Data: Sources (15); Sources (17)].
- Emergency geo/CA action: an emergency request was made to route “all users without MFA” through a geolocation control (countries listed and a single IP requested) with a small exception list; the record shows the request but does not prove whether a tenant CA policy or firewall rule was fully applied — however this is the operational action reported and explains travelling users being blocked if it was enforced [Data: Sources (4)].
- Non‑browser automation failures: background/service accounts often fail under tightened inspection/CA because they use legacy auth or token formats; one documented ADFS/client misconfiguration (EnableJWT + RequestMFAFromClaimsProviders) caused non‑browser integrations to fail while browser logins still succeeded — this maps closely to your integration that works in the UI but fails headless [Data: Sources (11); Sources (10)].

This combined sequence (phish → emergency geofence/CA + perimeter/WAF/template drift → missing header / token & CA mismatches → non‑interactive failures) matches the simultaneous symptoms you see. [Data: Sources (15); Sources (17); Sources (4); Sources (11)]
