Short answer / summary
- Most likely root cause: a mismatch between the SharePoint ACL (folder permissions) and the Sensitivity Label’s encryption group plus token caching. The folder had Unique Permissions and was encrypted with a sensitivity label that referenced a different Azure AD group than the SharePoint ACL; users with cached tokens could access temporarily until tokens refreshed, causing the random/partial behaviour (including the “works in Incognito” symptom) [Data: Sources (31, 30)].
- What to fix now: align the encryption group and the SharePoint permission group (or add required users to the label-encryption group), remove and reapply the label correctly, then invalidate user tokens so claims are refreshed. These exact remediation steps were applied successfully in the incident and restored stable access [Data: Sources (31)].
- Ticket level: escalate to 2nd-level (M365 / Identity / Information Protection) — this was the level that handled and resolved the issue in the recorded case [Data: Sources (30, 31)].

What the logs and investigation showed (evidence)
- The folder was renamed and had Unique Permissions applied, then a Sensitivity Label with encryption was applied afterwards. The label’s encryption was configured to allow access only for an Azure AD group different from the group referenced in the SharePoint ACL [Data: Sources (31)].
- Because the label was applied after the unique permissions and “Preserve custom permissions” was not enabled, SharePoint’s ACL still showed access allowed while the encryption policy required a different group membership. Depending on whether a user’s authentication token still carried the older claims, access could be granted or blocked, causing inconsistent results (including cases where Incognito worked temporarily) [Data: Sources (31, 30)].
- Tests showed adding a user directly to the label’s allowed group immediately granted access; removing them denied access again. Reproduction in a test tenant confirmed cached tokens allowed temporary access until token renewal [Data: Sources (31)].

Immediate remediation steps (what I recommend you do now)
1. Short-term (urgent)
   - Remove the Sensitivity Label from the folder or temporarily disable encryption until you can align group references for the label and the SharePoint ACL [Data: Sources (31)].
   - Add the affected users (or the SharePoint resource group) to the label’s allowed/encryption group if appropriate, or change the label to reference the same resource group that the SharePoint ACL uses [Data: Sources (31)].
   - Force token refresh for affected users (have them sign out of Office apps, browsers, and/or revoke refresh tokens via Azure AD) to eliminate cached claims and validate access consistently [Data: Sources (31, 30)].

2. Re-apply and verify
   - Reapply the Sensitivity Label after confirming the label’s allowed group matches the SharePoint permission group (or the group nesting) and retest with multiple accounts and devices (including one that had previously worked in Incognito) [Data: Sources (31)].
   - Monitor sign-in and access logs to confirm tokens and group claims are correct.

Preventive and governance actions (to avoid recurrence)
- Align the SharePoint permission model and labeling practice with your documented architecture: use dedicated SharePoint resource groups (not functional groups) nested consistently, avoid parallel groups with the same name, and document changes before setting Unique Permissions or applying labels [Data: Sources (24)].
- When applying a Sensitivity Label that enforces encryption, verify whether “Preserve custom permissions” or equivalent is needed and ensure the label’s encryption groups are identical to the SharePoint resource group object IDs (not just names) — SharePoint evaluates by object ID [Data: Sources (31, 24)].
- After permission or label changes, force token refresh for a test group and validate access, and include token-refresh steps in your change runbook.
