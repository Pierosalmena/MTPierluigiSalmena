1) The “/Projects/2025_Migration” folder: permissions suddenly denying users
- A prior case with the exact “folder worked yesterday, now access denied” symptom was caused by incorrect group nesting in Entra ID after a permission-group change. Fixing the nesting restored access [Data: Sources (20)].
- Practical implication: if the SharePoint permission shows access via a group that was recently re-created or renamed, users may have lost effective permissions because the nesting/links were not updated [Data: Sources (20)].

2) The renamed folder + “Confidential” label producing inconsistent behavior (some users, incognito works briefly)
- Renaming a folder and then applying a sensitivity label that enforces encryption / access by a different Azure AD group can create a conflict between SharePoint ACLs and AIP encryption (different groups or object IDs), leading to inconsistent, intermittent access [Data: Sources (30)].
- A related, separate pattern is when two groups with the same name (one cloud-native, one synced from on‑prem) exist — external guests in the cloud group can be excluded once Azure AD Connect starts overwriting/merging objects, producing exactly the sporadic “works in incognito / only after re-add / then fails again” behavior [Data: Sources (13)].
- Token caching and Conditional Access delays can also produce partial access where signing out / signing in, or running incognito, temporarily helps — this has been seen and was resolved by forcing re-authentication, triggering device sync or adjusting the CA policy for the affected resource [Data: Sources (38)].
