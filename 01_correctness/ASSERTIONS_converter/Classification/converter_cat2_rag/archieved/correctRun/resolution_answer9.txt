Summary — what the indexed records show (explicitly documented)
- The four symptoms you describe match a known set of incidents during KF70’s second migration wave: (a) cloud sign-in works but on‑prem apps fail because on‑prem UPNs were overwritten to a new suffix; (b) cloud group membership remained the old groups because the new groups were placed outside the AAD Connect OU filter; (c) built‑in mobile mail clients failed because tenant Modern Authentication was enforced and those clients tried Basic Auth; (d) scheduled tasks using service accounts failed because those accounts were included in the tenant Conditional Access (MFA) target group and were synced/licensed unexpectedly. These root causes and their fixes are documented in tickets TI-1700001, TI-1700002, TI-1700003 and TI-1700004 [Data: Sources (22, 53, 4, 49)].

Step‑by‑step resolution based on the documented fixes
Below are the concrete steps that were performed in the recorded incidents. If you want us to perform these for your environment we can follow the same sequence; where the records show actions taken I cite the ticket.

1) Fix “can sign into M365 but not on‑prem apps” (UPN suffix mismatch)
- Verify whether affected users’ on‑prem userPrincipalName was changed to the new cloud suffix (kf70group.local) while on‑prem apps still expect kf70.local — this mismatch was the documented root cause [Data: Sources (22)].
- If UPNs were overwritten, revert the on‑prem userPrincipalName values to the original kf70.local form for affected accounts (the record shows this was done for 25 users) [Data: Sources (22)].
- Trigger a full Azure AD Connect synchronization after the reversion to ensure on‑prem and cloud attributes align [Data: Sources (22)].
- Validate by testing 2–3 affected users against the on‑prem applications (the ticket validation step used 3 users before wider validation) [Data: Sources (22)].

2) Fix cloud/on‑prem group membership divergence (AAD Connect OU filter)
- Confirm whether the new group objects for the second wave were created in an OU outside the Azure AD Connect sync scope (the documented OU name was Migration_Phase2) [Data: Sources (53, 54)].
- Update the Azure AD Connect OU filter to include the OU containing the new group structure (the engineers added Migration_Phase2 to the sync scope in the record) [Data: Sources (53, 54)].
- Run an initial delta then a full sync to bring group objects and memberships into Entra ID and propagate member relationships [Data: Sources (53, 54)].
- Validate that affected users now show the intended group memberships in Microsoft 365 and that downstream access (SharePoint, apps) evaluates correctly [Data: Sources (53, 54)].

3) Fix mobile built‑in mail clients failing (Modern Auth / EAS Basic Auth blocked)
- Confirm that Modern Authentication is enforced tenant‑wide (this was enabled as part of the migration baseline) and that affected clients (iOS built‑in Mail, Android Gmail using EAS) are attempting Basic Auth [Data: Sources (4)].
- For affected users, remove the old cached credentials on the device and re‑add the account using a client/method that supports OAuth (Outlook mobile or updated built‑in client with modern auth). In the record they instructed customers to update the mobile clients to use modern auth and to re‑enroll affected devices in Intune where appropriate [Data: Sources (4)].
- Validate by confirming those devices can connect and that Outlook mobile continues to work for any users who use it [Data: Sources (4)].

4) Fix scheduled tasks failing with authentication errors (service accounts & Conditional Access)
- Check whether service accounts were moved into the tenant target group (KF70-AllUsers-Hybrid) that the baseline Conditional Access policy KF70-Require-MFA targets; sign‑in logs were used to confirm “Failed - Conditional Access requires MFA” in the documented case [Data: Sources (49, 50)].
- Create a dedicated exclusion group for service accounts (the record shows KF70-ServiceAccounts-NoMFA was created), add the service accounts to it, and add that group as an explicit exclusion to the Conditional Access policy so MFA is not applied to non‑interactive logons [Data: Sources (49, 50)].
- Re‑evaluate the Azure AD Connect filter to avoid unintentionally syncing service accounts (which led to license assignment and subtle auth path changes in the documented case) [Data: Sources (49, 50)].
- Validate by confirming scheduled tasks run successfully overnight (the customer confirmed success in the ticket) [Data: Sources (49, 50)].

Validation and documentation steps that were performed in the incidents
- After reverting UPNs the engineers triggered a full sync and validated a sample of users (3 users) before wider validation, then confirmed all 25 users could sign in to both cloud and on‑prem apps [Data: Sources (22)].
- After updating OU filters they ran delta/full syncs and the customer validated group memberships and SharePoint permissions [Data: Sources (53, 54)].
- After changing CA exclusions and AAD Connect scope, scheduled tasks ran successfully and the migration runbook was updated to require explicit service‑account handling [Data: Sources (49, 50)].
- Mobile client fix required user device action (remove/re‑add account, re‑enrol) and confirmation that Outlook mobile continued to work [Data: Sources (4)].
